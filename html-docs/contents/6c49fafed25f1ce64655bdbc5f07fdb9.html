Description

Diagram de Sequence du processus d'enregistrement des Athlètes

Le diagramme de sequence va nous permettre de décrire la dynamique des instances associées,

  • admin

    • ici notre admin qui intéragit avec notre application demande au systeme d'instancier un objet insert

  • insert

    • La classe InsertionAthlete instancie un objet insert et cette dernière fait appel à un service qui va lui retourner une liste d'athlètes à partir d'un path csvFile.

  • service

  • athleteDao

    • Les différentes étapes qui vont permettre aux athlètes d'être enregistrés en base de données se font par cette instance athleteDao. Elle agit comme un receveur des requêtes provenant de l'acteur et permet de réaliser les différentes opérations en base de données (DAO: Data Access Objet)

  • interface IathleteDao

    • Le contrat entre l'interface IatheleteDao et la classe AthleteDao vont permettre d'implementer les operations crud de l'interface iAthleteDao. Nous utilisons cette interface car elle peut etre réutilisée ailleurs dans le code pour procéder à des opérations en base de données.

Le service va faire appel aux méthodes CRUD de l'instance athleteDao et retourner une réponse selon la condition du status sur l'athlete si il est présent en base de données ou non:

  • Si l'athlète n'est pas présent dans la base isAthlete() retourne false, alors on peut l'ajouter en base de données
  • Si l'athlète existe en base de données on peut continuer les autres opérations comme updateAthlete
